# Deal Finder Post-1.0 Expansion Plan

This document replaces the earlier v1.5 review memo with an implementation-ready, phase-driven plan for evolving Deal Finder beyond `1.0.0`.

This plan is written for an AI implementation agent. It is intentionally explicit.

The goal is **not** to rewrite the existing plugin from scratch.
The goal is to **extract**, **stabilize**, and **extend** the current `1.0.0` architecture in a way that preserves existing OpenClaw value while adding:

- a localhost-capable standalone mode
- a non-technical web UI
- a future-safe persistence abstraction
- a path to richer distribution without weakening safety

---

## 1. Current Reality

Before changing anything, the agent must treat these as factual properties of the current codebase.

### 1.1 What already exists

The current system already has:

- a published `1.0.0` OpenClaw plugin
- 50+ working OpenClaw tools grouped across watch lifecycle, views, scanning, reporting, workflows, discovery, review, and diagnostics
- modular internal code under:
  - `src/tools/`
  - `src/lib/engine*.ts`
  - `src/lib/discovery*.ts`
  - `src/lib/report*.ts`
  - `src/lib/heuristics*.ts`
  - `src/lib/store*.ts`
- bounded URL safety:
  - scheme validation
  - host allow/block rules
  - localhost and private-IP blocking
  - DNS resolution checks
  - redirect revalidation
- bounded fetch behavior:
  - conditional GET support
  - shared undici dispatcher reuse
  - per-host pacing
  - byte caps
  - truncation reporting
- bounded same-product discovery:
  - manual candidate mode
  - Firecrawl search mode
  - no arbitrary recursive crawling
- optional review behavior:
  - `off`
  - `queue`
  - `auto_assist`
- JSON persistence with:
  - watch history
  - import provenance
  - review provenance
  - saved views

### 1.2 What does not exist yet

The current system does **not** have:

- a standalone HTTP API server
- a standalone admin UI
- a database backend
- a general repository abstraction
- a cross-runtime shared core package
- a notification delivery layer beyond current tool/report surfaces
- multi-user auth/session/tenant boundaries

### 1.3 Current architecture hotspots

The codebase is materially modular already. The remaining technical pressure points are:

- `src/lib/store.ts`
  - still owns too much live persistence behavior
- `src/lib/engine-result.ts`
  - large result-shaping and alert-classification module
- `src/lib/report-workflow-rankings.ts`
  - dense domain logic for prioritization
- `src/lib/report-insights.ts`
  - dense insight/provenance computation
- JSON store as the only runtime backend
- no shared service layer exposed outside OpenClaw tool adapters

These are the real starting points. The earlier “monolith everywhere” diagnosis is no longer accurate.

## 2. Program Goals

This program should deliver a phased expansion of Deal Finder while preserving the existing plugin.

### 2.1 Primary goals

1. Preserve the current OpenClaw plugin behavior and public tool surface.
2. Extract a shared core runtime that does not depend on OpenClaw APIs.
3. Add a standalone localhost server mode for non-OpenClaw usage.
4. Add a web UI for non-technical users who do not want to manage commands manually.
5. Introduce an optional database backend without breaking existing JSON users.

### 2.2 Non-goals for this program

The following are explicitly out of scope unless a later phase says otherwise:

- browser automation in the default runtime
- anti-bot or proxy-rotation features
- recursive or arbitrary crawling
- multi-tenant SaaS architecture
- PostgreSQL support
- email/SMS integrations
- ML price prediction
- coupon scraping systems
- public Internet-first deployment hardening

### 2.3 Release framing

The file name is `1.5-release.md`, but execution should be thought of as a **program of phased changes**, not one giant drop.

Do **not** force everything into a single release branch if the checkpoints are not green.

---

## 3. Non-Negotiable Invariants

The implementation agent must preserve all of the following unless the plan explicitly says otherwise.

### 3.1 Safety invariants

- URL safety must not weaken.
- All fetches must continue to honor:
  - `allowedHosts`
  - `blockedHosts`
  - private-IP blocking
  - localhost blocking
  - redirect validation
  - per-host pacing
  - global concurrency bounds
  - byte caps
- Discovery must remain bounded and explicit by default.
- Review must remain policy-driven and off by default.

### 3.2 Compatibility invariants

- Existing OpenClaw tool names must continue to work.
- Existing OpenClaw plugin installation path must remain valid.
- Existing JSON stores must remain readable.
- Existing import/export payloads must remain usable.
- Existing snapshot provenance and review provenance must not be dropped.
- Existing saved view semantics must remain intact.

### 3.3 Product invariants

- The project remains install-safe and community-plugin-friendly.
- The default runtime remains conservative.
- New standalone features must not silently backdoor unsafe scraping behavior into the plugin.

---

## 4. Architectural Direction

### 4.1 Target high-level shape

The system should evolve into three explicit layers:

1. `core`
   - runtime-agnostic business logic
   - no OpenClaw imports
   - no HTTP-framework imports
2. `adapters/openclaw`
   - current plugin registration and tool exposure
3. `adapters/standalone`
   - localhost server
   - web UI serving
   - optional real-time push

### 4.2 Important constraint

Do **not** immediately move the entire tree into a brand-new folder structure if that creates churn without value.

Prefer **incremental extraction**:

- extract services from current `src/lib/` modules
- keep the current code working
- move only once the new service boundaries are verified by tests

### 4.3 Recommended target structure

This is the target shape after multiple phases, not a day-one refactor:

```text
src/
  core/
    config/
    domain/
      entities/
      value-objects/
      policies/
    repositories/
    services/
    ports/
  adapters/
    openclaw/
    standalone/
      http/
      realtime/
      ui/
  infrastructure/
    persistence/
      json/
      sqlite/
    fetch/
    discovery/
    extraction/
    review/
  legacy/
    transitional-bridges/
```

This is a direction, not a hard prerequisite for Phase 1.

## 5. Delivery Strategy

This program must be executed with **phase-driven test-driven development**.

### 5.1 Mandatory TDD rule

For every phase:

1. define the contract
2. write or update tests first
3. implement the minimum code to make tests pass
4. refactor
5. rerun:
   - `npm test`
   - `npm run build`
   - any new phase-specific verification

### 5.2 Mandatory stop rule

Do **not** begin the next phase until:

- tests are green
- build is green
- compatibility checks for the current phase are green
- documentation for the current phase is updated

### 5.3 Mandatory branch discipline for AI agent

The agent must treat each phase as a checkpoint:

- do not start broad multi-phase edits in one pass
- do not partially scaffold future layers without current tests
- do not leave dead parallel implementations behind

## 6. Phase Plan

## Phase 0: Baseline Lock and Characterization

### Purpose

Freeze current behavior so future refactors can be judged safely.

### Deliverables

1. create or strengthen characterization tests for:
   - watch CRUD
   - saved views
   - scan result shaping
   - discovery bounded behavior
   - review policy behavior
   - import/export semantics
   - provenance persistence
2. document the current JSON store behavior in code comments or test fixtures
3. document current output-shape expectations for key tools

### TDD tasks

- add golden-style tests for representative tools where behavior must not drift
- add fixtures for representative JSON store versions and migrated loads

### Acceptance criteria

- current OpenClaw plugin behavior is test-characterized enough that extraction refactors can be judged against it

---

## Phase 1: Extract Shared Core Services

### Purpose

Introduce real service boundaries without breaking the plugin.

### Required new abstractions

At minimum:

- `WatchRepository`
- `SavedViewRepository`
- `WatchService`
- `SavedViewService`
- `ScanExecutionService`
- `ScanCommitService`
- `DiscoveryService`
- `ReviewService`
- `ReportingService`

### Requirements

1. interfaces must not import OpenClaw
2. current tool modules must delegate into services
3. JSON storage remains the live backend in this phase
4. no standalone server yet

### Suggested extraction mapping

- from `src/lib/store.ts`
  - repositories / persistence helpers
- from `src/lib/engine*.ts`
  - scan execution and commit services
- from `src/lib/discovery*.ts`
  - discovery service
- from `src/lib/report*.ts`
  - reporting service boundaries
- from `src/lib/llm-review.ts` and `src/lib/review-policy.ts`
  - review service + review policy layer

### TDD tasks

- add service-level tests for:
  - watch lifecycle operations
  - saved-view lifecycle operations
  - scan execution
  - scan commit reconciliation
  - discovery evaluation
  - report generation
  - review candidate selection

### Explicit anti-patterns

- do not make services depend on `OpenClawPluginApi`
- do not move everything into new folders before contracts exist
- do not duplicate business logic between old tool handlers and new services

### Acceptance criteria

- tool handlers still behave the same
- service tests exist and pass
- OpenClaw remains fully functional
- JSON backend remains canonical

## Phase 2: Repository Abstraction and JSON Backend Hardening

### Purpose

Make storage replaceable without breaking current users.

### Requirements

1. wrap current JSON persistence behind repository implementations
2. preserve current version-2 JSON shape
3. preserve:
   - import provenance
   - reviewed-field provenance
   - snapshot truncation metadata
   - saved views
   - history append semantics
4. harden corruption handling and startup validation

### Suggested interfaces

- `WatchRepository`
  - get/list/search/save/delete/bulk operations
- `SavedViewRepository`
  - CRUD for saved views
- `StoreMaintenancePort`
  - export/import/migration helpers

### TDD tasks

- tests for malformed JSON
- tests for partial store recovery behavior
- tests for migration from version 1 to version 2
- tests for import/export round trips

### Acceptance criteria

- JSON backend is an explicit adapter
- all existing plugin behavior remains green
- JSON remains the default backend

---

## Phase 3: Standalone Localhost API (Single-User First)

### Purpose

Add a localhost-capable distribution mode without dragging in multi-user complexity.

### Scope choice

For this phase, standalone mode is:

- single-user
- self-hosted
- localhost-first
- not multi-tenant
- not public-Internet-first

### Requirements

1. choose one framework:
   - **Fastify**
2. expose a REST API that maps onto existing domain capabilities
3. do not weaken OpenClaw support
4. no UI yet

### Endpoint families

Implement resource-oriented endpoints that map cleanly onto the current tool surface:

- `/api/v1/watches`
- `/api/v1/views`
- `/api/v1/scans`
- `/api/v1/discovery`
- `/api/v1/reviews`
- `/api/v1/reports`
- `/api/v1/system`

### Important rule

The REST API is **not** a replacement for the OpenClaw tool surface.
It is a second adapter over the same capabilities.

### Required API details

- canonical error envelope
- input validation
- stable response envelopes
- health endpoints:
  - `/health/live`
  - `/health/ready`
- request logging
- startup config validation

### Security posture for this phase

Choose one explicit mode and document it:

- default: localhost-only, no auth
- optional: token-based protection for non-localhost binding

Do **not** leave auth “optional and vague.”

### TDD tasks

- API contract tests
- service-to-API integration tests
- health endpoint tests
- config validation tests
- bad-input and error-envelope tests

### Acceptance criteria

- standalone server can run locally without OpenClaw
- existing OpenClaw plugin still works
- same core services power both adapters

## Phase 4: Web Dashboard for Non-Technical Users

### Purpose

Give users a way to manage Deal Finder without issuing OpenClaw commands manually.

### Build choice

Use:

- React 18
- Vite
- Tailwind CSS
- TanStack Table

### Serving model

- build static assets with Vite
- serve them from the Fastify adapter under `/ui`
- no SSR in the first version

### Minimum viable UI

Do **not** try to ship the final dream dashboard in one phase.

Phase 4 UI scope:

1. dashboard home
   - total watches
   - active alerts
   - recent changes
   - top opportunities
2. watch management
   - list/search/filter
   - add/edit/remove
   - enable/disable
   - tags/groups
3. saved views
   - list/create/update/delete/run
4. scan actions
   - trigger scan
   - inspect recent scan results
5. history and insights
   - per-watch history
   - trends
   - insights

### Explicitly defer from first UI phase

- complex arbitrage visualization
- discovery workbench
- advanced keyboard power mode
- drag-and-drop organization
- full no-JS progressive mode

### TDD tasks

- UI component tests
- API contract tests for UI consumption
- end-to-end tests for:
  - create watch
  - edit watch
  - run scan
  - create/run saved view
  - inspect report

### Acceptance criteria

- a non-technical user can add a watch, run a scan, and review results entirely via the web UI

## Phase 5: Optional SQLite Backend

### Purpose

Add a safer growth path for persistence without breaking current JSON users.

### Key rule

SQLite is **opt-in first**, not a forced switch on day one.

### Requirements

1. define schema for:
   - watches
   - saved views
   - snapshots/history
   - provenance metadata
2. preserve the same observable behavior as the JSON backend
3. keep JSON readable and supported for at least one transition release

### Recommended technology choice

- SQLite
- Kysely or Drizzle

Pick one explicitly before coding.

### Migration contract

- JSON backup is mandatory
- migration is offline-only in the first release
- migration verification is mandatory
- rollback story is documented
- no dual-write mode in the first iteration unless fully tested

### TDD tasks

- backend parity tests: JSON vs SQLite
- migration tests
- round-trip export/import tests
- large-history behavior tests

### Acceptance criteria

- SQLite backend passes the same behavior contracts as JSON
- migration is non-destructive and auditable

## Phase 6: Real-Time Updates

### Purpose

Add live UX only after API and UI basics are stable.

### Recommendation

Start with **SSE** unless bidirectional behavior becomes necessary.

If WebSockets are used, the prompt must define:

- authentication
- authorization
- subscription scope
- idle timeout
- connection limits
- reconnect semantics

### Suggested events

- scan completed
- alerts updated
- watch updated
- review queue changed

### TDD tasks

- event serialization tests
- auth/connection tests
- reconnection tests
- rate-limit tests

### Acceptance criteria

- users can see live updates without page refresh
- transport does not weaken security posture

## 10. Things the AI Agent Must Not Do

- Do not rewrite the codebase from scratch.
- Do not collapse the OpenClaw tool surface into “just 8 resources.”
- Do not introduce Playwright or browser automation into the default runtime.
- Do not remove JSON support before SQLite parity is proven.
- Do not weaken URL safety or discovery bounds.
- Do not add multi-user auth complexity unless explicitly chosen.
- Do not implement vague “optional” security. Make runtime security modes explicit.
- Do not start UI work before the standalone API boundary is stable.

---

## 11. Success Criteria for the Program

This program is successful when all of the following are true:

1. OpenClaw mode still works with backward-compatible behavior.
2. A shared core exists that does not require OpenClaw imports.
3. A localhost standalone server can perform core Deal Finder workflows.
4. A non-technical user can manage watches through a UI.
5. JSON users are not broken.
6. SQLite, if introduced, is optional first and migration-safe.
7. Safety posture is equal to or stronger than `1.0.0`.

---

## 12. Final Instruction to the Implementation Agent

Do this in phases.

Do it test-first.

Do not chase the final ideal architecture before stabilizing each compatibility layer.

The correct mindset is:

1. preserve current value
2. extract reusable core boundaries
3. add standalone API safely
4. add UI on top of stable API
5. add optional persistence upgrades
6. keep safety and compatibility stronger than convenience
